Tessell Internship

Overview

AI Intern · Tessell · April 2026 – Present

Tessell is an AI-native, multi-cloud database-as-a-service platform that helps companies provision, monitor, and manage their cloud databases from a single control plane. Before I officially started, Tessell gave me a pre-assignment for the AI Internship: design and build an AI-Powered Data Platform Assistant, a tool that could look at a database system's own performance data and flag problems before they became real issues. I am still interning at Tessell as of this writing.

What I Built

  • Built a data analysis tool using Python, pandas, SQLite, scikit-learn, and Streamlit to help review system performance data.
  • Developed an anomaly detection model using Isolation Forest to identify unusual patterns in CPU, memory, disk, and network data.
  • Created project documentation and presentation materials explaining the AI workflow, data pipeline, and user interface.

The idea was to give engineers a fast way to answer "is something wrong with this system, and where," without needing to manually dig through raw metrics. The Streamlit interface reads system performance history out of a SQLite store, an Isolation Forest model flags time windows that look statistically abnormal across CPU, memory, disk, and network usage, and the tool surfaces those flagged windows back to the user instead of a flat wall of numbers.

The sidebar breaks the tool into five sections: Load Data, SQL Query, Anomaly Detection, Ask AI, and a REST API Demo. In the screenshot above, asking "Show me the anomalies" generates SELECT * FROM metrics WHERE anomaly = -1; against the stored performance data, and the assistant explains the result in plain English rather than just dumping a table.

Representative Code

The anomaly detection side trains an Isolation Forest on the system's core performance columns, then stores a simple +1 or -1 flag per row so the rest of the app can query for anomalies directly in SQL:

# Representative of the anomaly detection logic, reconstructed for this page. import pandas as pd from sklearn.ensemble import IsolationForest import sqlite3 FEATURES = ["cpu_usage", "memory_usage", "disk_usage", "network_requests", "db_connections"] def flag_anomalies(df, contamination=0.05): model = IsolationForest(contamination=contamination, random_state=42) # -1 marks an anomaly, 1 marks a normal row, same convention scikit-learn uses df["anomaly"] = model.fit_predict(df[FEATURES]) return df def save_to_sqlite(df, db_path="metrics.db"): conn = sqlite3.connect(db_path) df.to_sql("metrics", conn, if_exists="replace", index=False) conn.close()

Every row of uploaded system performance data gets scored this way before it lands in SQLite, so anomalies are just another column you can filter on.

The "Ask AI" section takes a plain-English question, maps it to a SQL query template, runs it against the SQLite database, and hands the result to an LLM to explain in plain English:

# Representative of the Ask AI to SQL flow, reconstructed for this page. def answer_question(question, conn): question_lower = question.lower() if "anomal" in question_lower: sql = "SELECT * FROM metrics WHERE anomaly = -1;" elif "cpu" in question_lower: sql = "SELECT * FROM metrics ORDER BY cpu_usage DESC LIMIT 10;" else: sql = "SELECT * FROM metrics ORDER BY timestamp DESC LIMIT 10;" result = pd.read_sql_query(sql, conn) explanation = explain_with_llm(question, result) return sql, result, explanation

A real deployment would route this through a proper LLM prompt instead of keyword matching, but the shape of the pipeline, question in, SQL and explanation out, is the same one shown in the screenshot above.

Watch It Run

Watch on YouTube

The robot in this run used a six motor drivetrain, with the remaining two motors split between a pneumatic catapult and an intake. A vision sensor mounted on the turret end identified game objects and their field position, odometry tracked the robot's own position continuously, and a PID controller handled every drive and turn command so the motion stayed smooth and accurate with zero human input.

How the Robot Made Decisions

The vision sensor returns a list of detected objects each frame, tagged by color signature along with their position and size in the camera's field of view. From there the pipeline was straightforward in concept and demanding in execution: pick the best target, turn the drivetrain to face it using the same PID turn loop from our VRC robots, drive to it using odometry to confirm arrival, intake it, then repeat until the field runs dry or time runs out.

The original code for this robot lives on a computer we do not have on hand, so what follows is a rebuilt example in the same spirit and style, not a byte for byte copy, showing the kind of object selection and pneumatic firing logic that ran the robot in this video.

// Illustrative reconstruction of our VEX AI decision loop. // Original competition code is on another machine; this shows the same approach. struct DetectedObject { int signature; double angleOffset; // degrees from center of camera FOV double distance; // estimated inches to object }; std::vector<DetectedObject> getVisibleObjects() { std::vector<DetectedObject> objects; auto detections = aiVision.get_objects(); for (auto& obj : detections) { if (obj.signature == GAME_OBJECT_SIG) { DetectedObject d; d.signature = obj.signature; d.angleOffset = (obj.x_center - CAMERA_CENTER_X) * DEGREES_PER_PIXEL; d.distance = estimateDistance(obj.width); objects.push_back(d); } } return objects; } // Picks the closest valid object rather than the first one seen DetectedObject pickBestTarget(std::vector<DetectedObject>& objects) { DetectedObject best = objects[0]; for (auto& obj : objects) { if (obj.distance < best.distance) { best = obj; } } return best; } void aiAutonRoutine() { while (matchTimeRemaining() > 0) { auto objects = getVisibleObjects(); if (objects.empty()) { searchPattern(); continue; } DetectedObject target = pickBestTarget(objects); // reuse the same PID turn loop built for VRC, now driven by vision error pidController.pidTurnToAngle(target.angleOffset); // odometry confirms we actually arrived before intaking odometry.driveToRelative(target.distance); intake.spin(fwd, 100, pct); pros::delay(300); if (readyToScore()) { catapultSolenoid.set(true); // fire the pneumatic catapult pros::delay(200); catapultSolenoid.set(false); // retract for the next cycle } } }

A rebuilt example of the vision to PID to pneumatics pipeline: see an object, turn to it with the same PID controller from our VRC robots, confirm arrival with odometry, intake, then fire the pneumatic catapult to score.

The interesting part is how little of this is actually new. The vision sensor supplies a target angle instead of a joystick input, but the PID turn loop underneath is the exact same math we tuned for VRC. Odometry answers the same "where am I" question it always did. The only genuinely new piece is the perception layer that decides what to aim at in the first place, everything downstream of that decision reused code we had already proven on the competition field.

2-Speed Pneumatic Transmission

Shift-on-the-Fly Gearing for a Competition Drivetrain

Team 3303D · VRC Tipping Point

Most VEX drivetrains pick one gear ratio and live with the tradeoff all season. We refused. Our drivetrain could shift between two gear ratios mid-match: a torque mode for climbing the platform, and a speed mode for rushing mobile goals during autonomous, often the play that decides the match.

Why Pneumatics

We needed something to physically slide a gear a fixed distance, instantly, on command, without tying up a motor. Pneumatics fit perfectly. A VEX pneumatic system stores compressed air in a small onboard tank, charged by a pump before the match, and releases it through a solenoid valve into a piston cylinder. Flip the valve and the piston fires in a fraction of a second with no ramp up, no encoder feedback loop, and no ongoing motor draw once it has fired. That is exactly the behavior a two state gearbox needs: shift, hold, shift back, and none of that competes with the motors already busy driving the robot.

The tradeoff is that pneumatics are binary. A piston is either fully extended or fully retracted, there is no in-between speed control like a motor gives you. For shifting between exactly two gear ratios that limitation is actually a feature: it is impossible for the mechanism to get stuck half shifted, since the piston only knows two positions and the spring-loaded valve holds it there until air pressure changes again.

How It Works

The mechanism is a sliding-gear gearbox actuated by a piston. We built a custom linear slide, a spacer wrapped around a c-channel so it can glide along it, and mounted the piston to that slide. The shifting gear rides along with it: when the piston extends or retracts, it shoves the gear left or right along its shaft, meshing it with either the speed gear train or the torque gear train. Mirrored on both sides of the drive, all four motors stay engaged in both modes, only the gear ratio between the motors and the wheels changes.

In practice, one button on the controller fired the solenoid. In torque mode, the extra gear reduction let the drivetrain climb the platform without stalling. In speed mode, the direct higher ratio let us close distance across the field fast enough to win the race for mobile goals during autonomous. Because the shift happened in the time it takes compressed air to move a piston, the driver could change modes mid-match without losing more than a fraction of a second.

The Engineering Lesson

The transmission worked, but it cost us weight and drivetrain friction, and later in the season we made the hard call to remove it in favor of a lighter direct drive, freeing space for a better mobile-goal intake. Knowing when to kill your own clever design is as much an engineering skill as building it.

Disc Launcher

Two Generations of Disc Launchers

Team 3303C · VRC Spin Up

Spin Up was a disc shooting game, and over the season we engineered two complete launcher systems: a flywheel first, then a slip gear catapult. Each one was designed in CAD, backed by physics on the whiteboard, built, tested, and iterated. This page covers both, plus the math that made them work.

Generation 1: The Flywheel

Our first launcher stored kinetic energy in a spinning mass and transferred it to each disc on contact. The clever part was the power delivery: the flywheel shares its two motors with the intake through a ratchet system. Spinning one way, the motors drive the intake. Spinning the other way, the ratchet locks and all the power flows into the flywheel. Two mechanisms, full power to each, zero extra motors spent.

Watch the Flywheel Fire

The flywheel worked, and it helped us store kinetic energy between shots. But it had a weakness built into its physics: every shot drains the wheel, so rapid multi-disc volleys meant waiting for it to spin back up, and shot speed varied with how much energy was left. We wanted to launch a full stack of discs at once, consistently.

Generation 2: The Slip Gear Catapult

The catapult solved both problems with a beautifully simple mechanism: the slip gear, a gear with some of its teeth shaved off. Three gears, one motor. The driven gear connects to the motor and drives the slip gear. The slip gear meshes with the catapult gear, winding the arm down against elastic tension. The moment the shaved section rotates into place, the gears disengage and the arm snaps forward, launching every disc loaded on it at once. The same geometry doubles as overload protection: if the catapult jams, the slip gear slips instead of stalling the motor or snapping an axle.

Watch the Catapult Fire

The Math That Made It Work

A catapult that launches two disc and three disc stacks needs consistent trajectories for both, and foam discs do not fly like textbook projectiles. We worked the problem on the whiteboard in three parts.

1. Impulse per stack. Starting from J = F·Δt = m(v′ − v₀), we computed the impulse each stack needs in both axes. For a two disc stack (m₁ = 0.13 kg) that came out to roughly 0.112 N·s vertically and 0.097 N·s horizontally. For a three disc stack (m₂ = 0.195 kg) the numbers grow to about 0.168 N·s and 0.293 N·s. That told us exactly how much stored energy the elastics had to deliver in each case.

2. Drag. Our first trajectory predictions used plain kinematics and they missed, because air resistance on a light foam disc is not negligible. We rederived the flight starting from Newton's second law with a drag force proportional to velocity, separated variables, and integrated both sides. The result is an exponential decay law for the disc's velocity, v_f = v₀e^(−(b/m)T), which finally matched what we measured on the field and let us tune launch angle and elastic loading analytically instead of by guessing.

3. The elastics. The launch energy comes from rubber bands, so we measured them like a spring. Hanging known weights and plotting force against stretch gave a clean line with slope 1.4564 N/cm, which works out to a spring constant of about 146 N/m for our blue bands. With k known, choosing how many bands and how much pretension became arithmetic instead of trial and error.

The Outcome

The catapult launched full stacks in one motion with repeatable trajectories for both stack sizes, and the slip gear made it mechanically self protecting. Flywheel to catapult is the arc we are proudest of from that season: build the thing, measure honestly, do the physics, and let the numbers pick the winner.

Odometry & Custom PID Library

Making Robots Know Where They Are

Teams 3303C & 3303D · C++ (PROS & VEXcode)

Winning autonomous periods comes down to one thing: the robot knowing exactly where it is at all times, and getting there smoothly. We built that entire stack ourselves across two seasons: PID motion control, full field odometry, and custom drive curves, all documented meeting by meeting in our engineering notebooks. This page walks through all of it: the code, the math, the graphs, and how each idea led to the next.

It Started With a Problem

After our first competition we realized the stock VEX movement functions were not accurate enough. Turns drifted, drives overshot, and every routine needed hand tuned delays. So we learned PID and wrote our first controller. Proportional handles the current error, integral accumulates error over time to remove steady state offset, and derivative reacts to the rate of change to damp overshoot. Our very first PID function, a turn controller reading the inertial sensor:

int turnPID(){ while(enableTurnPID){ int heading = inertial.get_heading(); desiredTurnValue -= heading; error = desiredTurnValue - heading; derivative = error - prevError; totalError += error; prevError = error; double turnPower = (error*kP + derivative*kD + totalError*kI); rightBack.moveVoltage(-turnPower); rightMid.moveVoltage(-turnPower); rightTop.moveVoltage(-turnPower); leftBack.moveVoltage(turnPower); leftMid.moveVoltage(turnPower); leftTop.moveVoltage(turnPower); std::string HEADING = std::to_string(heading); pros::lcd::set_text(1, HEADING); pros::delay(20); } return 1; }

Our first PID controller, from the 3303C notebook. It worked, but it was slow and rough.

To make it good we tuned the constants with a disciplined process: zero out Ki and Kd, raise Kp until the system oscillates, add Ki until steady state error disappears, add Kd until overshoot is gone, then repeat until it converges. Tuning fixed the sluggishness, but a second problem surfaced: if the robot never quite reached its target, the loop ran forever and the robot froze mid autonomous.

Iteration Two: Timeouts and Velocity Caps

That failure pushed us to research PROS, the C++ framework from Purdue, and its Okapi library. We rebuilt the controller around an iterative position PID with two safety ideas of our own: a timer so the loop always exits even if the target is never perfectly reached, and a max velocity parameter so we could cap speed for delicate movements:

void driveForward(double distance, int ms, double maxVelocity) { IterativePosPIDController drivePID = IterativeControllerFactory::posPID(0.0014, 0.0002, 0.00015); drivePID.setTarget(distance); double initLeftPos = leftMid.getPosition(); double initRightPos = rightMid.getPosition(); int timer = 0; while (timer < ms) { double currentLeftPos = leftMid.getPosition(); double currentRightPos = rightMid.getPosition(); double vel = drivePID.step((currentLeftPos + currentRightPos) / 2); if (vel > maxVelocity){ vel = maxVelocity; } drive -> getModel() -> tank(vel, vel); pros::delay(10); timer += 10; } rightTop.setBrakeMode(AbstractMotor::brakeMode::hold); rightMid.setBrakeMode(AbstractMotor::brakeMode::hold); rightBack.setBrakeMode(AbstractMotor::brakeMode::hold); leftTop.setBrakeMode(AbstractMotor::brakeMode::hold); leftMid.setBrakeMode(AbstractMotor::brakeMode::hold); leftBack.setBrakeMode(AbstractMotor::brakeMode::hold); drive -> getModel() -> tank(0, 0); }

The rebuilt drive function: PID plus a timeout and a velocity cap. No more infinite loops.

One more usability fix: the function took motor degrees, which nobody thinks in. Using the wheel circumference and gear ratio we derived the conversion to inches and wrapped it:

void moveF(double dist, int m, double maxV) // inches { tareEncoders(); driveForward((dist * 1057), m, maxV); }

1057 encoder ticks per inch, so autonomous routines read in real units.

The PID Class

On 3303D we organized everything into a proper C++ class, the way industry codebases do it. Odometry: Knowing Where You Are

PID gets you to a target, but the target is only meaningful if you know where you started. That question, how does a robot know its position, became our odometry project. We evaluated the options in the notebook first: motor encoders alone skip and drift, two tracking wheels with an inertial sensor is simple but loses position when you get pushed, and three unpowered tracking wheels can reconstruct full position from geometry alone. We chose three tracking wheels fused with the inertial sensor.

The core insight is that between updates the robot travels in an arc. From the three encoder deltas you can compute the change in heading, and then the chord of the arc gives the true translation:

ΔD = 2·sin(θ/2) · < ΔS/Δθ + SS, ΔR/Δθ + SR >

The S-Curve: Rethinking Driver Control

The same obsession with smooth motion applied to driver control. The stock approach is linear: the joystick reads in the range [-127, 127] and maps straight to velocity, y = x. The problem is at the bottom of the range. Small joystick inputs still produce a lot of speed, so lining up a precise shot took forever.

Back on 3303D we had already fought this problem once with lookup tables: two hand built 128 entry arrays, trueSpeed and halfSpeed, that remapped every joystick value to a smoother acceleration profile because the old motors stalled on sharp inputs. That gave us the idea, but arrays are tedious. On 3303C we wanted a single equation.

Our first attempt was an exponential curve. It starts slow, giving fine control near zero, and ramps up toward full speed:

y = 1.2(1.043)x − 1.2 + 0.2x

The driver was happier but still wanted something better: fine control at low speed without giving up responsiveness in the middle of the range. Differential Power Split

Three Outputs, One Motor Budget

Team 3303D · VRC Tipping Point

VEX robots get a hard limit on motors, so every motor spent on one mechanism is a motor unavailable for another. Partway through the season our robot needed to run a conveyor belt and two separate four bar lifts, three independent outputs, but we only had the motor budget for a fraction of that if we built it the conventional way.

We started by considering just a differential, which splits one gearbox into two outputs. That solved two of the three needs, but not all three. Then we looked back at the two-speed transmission we had already built earlier that season and realized we could combine the two ideas: a differential for the first split, and a transmission acting as a power take off, or PTO, to further share the second output between both four bars. Run one system, and the gears spin one direction. Shift the power, and the same gears reverse and drive the other side instead.

We weighed it against the simpler option of just adding a conveyor belt on its own motors:

ProsCons
Conveyor BeltVery consistent, carries multiple rings, uses fewer motorsNot very efficient, not very fast, would not keep pace with other teams
Differential + PTOUses the motors to their fullest, enables true power transfer, frees up motors for other tasksHard to build, prone to failure

We picked the harder, more capable option.

Designing and Machining the Differential

We researched real world differential designs, then modeled our own custom differential in SolidWorks, sized to fit two motor mounts on a shared plate. Instead of 3D printing it, we cut it out of polycarbonate on the X-Carve, a CNC router, which gave us a stiffer, more precise part than we could get from a standard VEX plate.

Assembly and First Prototype

Once the plates were cut, we assembled them into a working prototype with motors mounted directly to the differential housing.

The first prototype told us a lot, both good and bad:

  • Did well: could cycle between the balls of a goal, and could move quickly between goals.
  • Could improve: the ball cycling speed needed work, and it struggled to score consistently once it got there.

We used that feedback to move into full integration rather than scrapping the concept.

Building the PTO and Wiring It In

With the differential proven, we built the PTO transmission that shares the second output between the two four bars, and connected gears from both four bars to the center of the robot where the differential lives.

One lesson carried straight over from the first transmission we ever built: shaving the gears slightly with a drill and a file, spinning the gear against the drill to flange the teeth, made the PTO mesh far more smoothly when it shifted. Tested with a piston, the transmission shifted cleanly between the two sides.

While we were in there, we cleaned up the whole electrical system too. The battery moved underneath the drivetrain next to the brain, and every wire got traced neatly and bundled with zip ties along the interior of the C-channel so nothing could snag or tear loose mid match. We also ran into a sizing problem: the shooter's sprockets were too wide for the robot to stay within the size limit, and they caught on the hood that directed the balls. The fix was to shave down one side of each sprocket, just enough to clear the hood while still leaving the rubber bands room to grab the balls.

Tuning and Protecting the System

In the final integration pass we repositioned the PTO to sit directly on the axle, which maximizes the force it transmits and minimizes losses along the way. We also added a back plate over the differential to protect the spinning motors from collisions and stray parts during a match.

The Payoff

One gearbox, three outputs, chosen mechanically instead of by spending more motors: the conveyor runs continuously off the differential's first output, and a single PTO shift decides which four bar gets the second. It remains one of the projects that taught the deepest mechanical lesson of the whole season, that cleverness in the gear train can buy capability the motor limit would otherwise refuse to give you.

Automotive-Style Differential (CAD Practice)

Learning Gearing by Modeling the Real Thing

Personal Project · SolidWorks

This one was not for a competition or a client, it was pure curiosity. After spending a season building VEX differentials out of small plastic gears, we got fascinated with how the real automotive version works: the bevel gear differential that lets a car's rear wheels spin at different speeds through a turn while still both receiving power from the same driveshaft. We wanted to actually understand the mechanism, not just use the VEX approximation of it, so we modeled a full scale style automotive differential from scratch in SolidWorks.

What's Actually Modeled

The assembly includes the large ring gear that receives input from the driveshaft's pinion gear, the differential case that the ring gear bolts to, a spider gear carrier with bevel pinion gears mounted inside, and the two side bevel gears that connect to independent axle shafts. When the car drives straight, the whole carrier spins as one unit and both axles turn at the same speed. When the car turns, the outer wheel needs to travel farther and spin faster than the inner wheel, and that is where the spider gears earn their keep: they roll around the stationary side gears, letting the two axle outputs split speed unevenly while still both receiving torque from the same ring gear.

Getting the bevel gear meshing to look and behave correctly in CAD, matching pitch cones so the ring gear, pinion, and spider gears all mesh at the correct angles, took real trial and error. It is a much more elegant solution than anything we had built with VEX parts, and modeling it by hand was the best way to actually understand torque and speed splitting instead of just taking it on faith from a textbook diagram.

Why It Mattered

This project has no scoreboard and no deadline, which is exactly what made it valuable. It let the differential idea we had already used competitively in the Power Split project connect to the real engineering behind it, and it is the kind of CAD practice that sharpens the skills that show up everywhere else on this page: careful mate relationships, gear ratio math, and patience with a model that will not quite mesh right until every dimension lines up.

Worm Gear Design

Locking a Lift Without a Ratchet

Team 3303D · VRC Tipping Point

Tipping Point scored points for controlling mobile goals, which meant the four bar lift on our robot spent most of a match holding a goal up in the air. Other teams knew that, and they would drive in and try to pry or shove a goal out of our lift to steal it. We needed the lift to stay exactly where we put it, under load, without us doing anything to hold it there.

Why a Worm Gear Instead of a Ratchet

The first idea most people reach for here is a ratchet: let the lift climb, then lock it against a pawl so it can't fall or get pushed back down. We considered it, but a worm gear did the same job better for us in two specific ways.

  • It still moves in both directions. A ratchet only lets you go one way. If we needed to lower the lift back down mid match, a ratchet would fight us. A worm gear drives the output shaft in either direction anytime the motor turns, so raising and lowering stayed fully under our control.
  • No pneumatics needed to disengage it. Disengaging a ratchet pawl generally means adding a pneumatic piston, a solenoid valve, and tubing, all just to let the mechanism move backward again. A worm gear does not need any of that. It is mechanically self locking in the sense that the load cannot back drive the worm, but the motor can always turn it, no piston required.

Mechanically, the reason this works is the same reason worm drives get used for holding brakes and jacks in industry: the pressure angle on a worm mesh means the gear can turn the worm's output shaft, but the output shaft essentially cannot turn the worm back. Once the motor stopped, the goal stayed locked in place, and the other team could push, ram, or shove all they wanted.

The Honest Tradeoff

Worm gears are not free performance. VEX motors are already fairly weak, and worm meshes are mechanically inefficient by nature, most of the motor's effort goes into overcoming friction in the mesh rather than lifting the load. The locking held perfectly and the goal never got stolen once this was installed, but the lift itself moved slowly, and gearing it differently did not fix that. We traded lift speed for a mechanism that could not be beaten by another robot just pushing on it, and for our matches that trade was worth it.

How the Software Works

The tuner's brain is a Raspberry Pi 4. The user selects an instrument and target note through the button interface, plays the note, and the device answers through the speaker: a confirmation ding if the note is within 0.5 Hz, or a spoken cue, "too sharp" or "too flat", if it isn't. Detecting pitch that precisely from a live microphone is the interesting part. A raw FFT of a short audio window only gives frequency resolution of several hertz, nowhere near our 0.5 Hz spec. We got around that with a windowed FFT followed by parabolic interpolation: fitting a parabola through the three FFT bins around the peak and solving for the true maximum, which lands between bins.

# Core pitch-detection pipeline (Raspberry Pi + USB microphone) import numpy as np import sounddevice as sd SAMPLE_RATE = 44100 WINDOW_SIZE = 8192 # ~0.19 s of audio per analysis frame TOLERANCE_HZ = 0.5 # spec: tune within 0.5 Hz def detect_pitch(samples): # Hann window to reduce spectral leakage windowed = samples * np.hanning(len(samples)) spectrum = np.abs(np.fft.rfft(windowed)) freqs = np.fft.rfftfreq(len(windowed), 1 / SAMPLE_RATE) peak = np.argmax(spectrum) if peak == 0 or peak == len(spectrum) - 1: return freqs[peak] # Parabolic interpolation around the FFT peak: # fits a parabola through (peak-1, peak, peak+1) and finds its vertex, # recovering the true frequency *between* FFT bins a, b, c = spectrum[peak - 1], spectrum[peak], spectrum[peak + 1] offset = 0.5 * (a - c) / (a - 2 * b + c) return freqs[peak] + offset * (freqs[1] - freqs[0]) def tune_loop(target_hz): while True: audio = sd.rec(WINDOW_SIZE, samplerate=SAMPLE_RATE, channels=1, dtype="float64") sd.wait() pitch = detect_pitch(audio[:, 0]) error = pitch - target_hz if abs(error) <= TOLERANCE_HZ: play_sound("in_tune.wav") # confirmation ding break elif error > 0: speak("too sharp") else: speak("too flat")

The analysis loop: record ~0.2 s of audio, window it, FFT it, refine the peak with parabolic interpolation, compare against the target note, and answer audibly.

Around this core, we built the state machine from our design flowchart: one click powers the device on, a second click starts tuning, and after two minutes of inactivity it shuts itself off to save battery. Note frequencies come from a lookup table of standard tunings for every instrument ISBVI carries, so the same device serves a violinist and a tuba player. We prototyped the pipeline in Python and moved the real-time loop toward C++ for snappier response on the Pi.

2.2 Functionality of System

The mobile robot system operates through several integrated subsystems, including custom lid attachments for secure tool storage, claws for precise tool handling, z-rails and z-motors for vertical movement, and advanced navigational capabilities for safe and efficient movement around the operating room. Before a scheduled surgical procedure, the robot deploys from its docking station, unlocking and unsealing its custom lid attachment and claw. It heads to a designated "parking zone" by the surgical bed, initializing the procedure protocol on its main screen. The robot can navigate the room with humans, entering "operation mode" when medical personnel are ready. It displays a timer, procedure details, tool inventory, and expected usage flow. Surgeons request tools via voice commands, and the robot retrieves them precisely. It effectively performs surgical technician duties while monitoring time, flow, and inventory. After surgery, it returns to the docking station for charging. Once docked, the station initiates disinfection procedures. The claw aligns with the robot using rails and motors, and elevators extend for cup collection. The claw, cups, and custom lid attachment lock in place for ultrasonic cleaning, which uses high-frequency sound waves to thoroughly remove debris from the tools. Subsequently, they unlock for mechanical cleaning, and then lock again in autoclave sanitation. After these procedures, the claw returns to the robot, elevators lift the sterilized equipment, and the custom lid attachment seals, ensuring tool safety until the next deployment.

2.5 Hardware Design Description

The robotic essential surgical technology (REST) system comprises a mobile robot and a docking station, each designed with meticulous attention to detail to enhance surgical procedures and instrument management. The mobile robot's structure includes a stainless steel base frame and an aluminum body frame, forming a robust skeleton covered by a protective polycarbonate frame. Four swerve modules at its corners constitute the drivetrain, each featuring drive and steering motors, gears, encoders, and billet wheels, which are wheels milled out of a chunk of solid material. The aluminum body frame integrates the electronic components, elevator system, and turntable. A screen module atop the frame displays vital information, while drive sensors and a charging dock ensure seamless mobility.

The docking station comprises a stainless-steel body frame housing essential cleaning and maintenance mechanisms. An ultrasonic cleaner, which includes a transducer (a device that converts electrical energy into high-frequency sound waves), generator, and temperature controls, uses these sound waves to thoroughly clean tools. A mechanical cleaner, with a stainless steel rack, spray arm, and heat element, uses high-pressure water and detergents to remove any remaining contaminants. An autoclave provides sterilization by generating pressure and steam to kill bacteria, viruses, and spores. The station's collection mechanism employs linear rails for multi-directional motion and is equipped with custom attachments to handle tools precisely. The housing accommodates a control panel for system operation and a charging dock receiver for the mobile robot. Together, the mobile robot and docking station create a sophisticated platform for surgical assistance, combining mobility, instrument care, and operational support.

2.7 Assembly of Innovation

The assembly of the Robotic Essential Surgical Technologist (REST) system involves two main components: the Mobile Robot (MR) and the Docking Station (DS). To assemble the MR, start by attaching four swerve modules to the stainless-steel base frame for omnidirectional mobility and connect the charging dock. The robot's ability to move smoothly and precisely is crucial in a surgical environment where space is limited, and precision is essential. Swerve modules enable the robot to navigate around obstacles and position itself accurately. Secure the aluminum frame to provide structural support and place electronic components, including a computer, power supply, control system, and wiring. Assemble the turntable mechanism separately and add it to the top of the chassis along with small roller bearings. Attach the screen and information module and encase the robot in a polycarbonate housing, adding operational components like the operation panel, status lights, speakers, and drive sensors. Ensure the charging dock receiver is connected for recharging and place a removable cover plate on top.

Couple the stainless-steel body frame with the base frame to form the chassis and attach the charging dock receiver for the DS. Secure the ultrasonic cleaner, mechanical cleaner, and autoclave modules, each with its components, to the main body frame. The three-step cleaning process ensures comprehensive decontamination and sterilization of surgical instruments. This multi-faceted approach, crucial for preventing infections and ensuring patient safety, thoroughly removes debris, contaminants, and pathogens. It represents the gold standard in surgical instrument cleaning, supporting the rigorous demands of modern surgical practices and helping to mitigate the risk of postoperative complications. Add the collection mechanism with linear rails, motors, carriage, custom claw attachment, and custom lid attachment. Finally, mount the control panel on top of the steel housing. Once assembled, the REST system is ready for surgical procedures, with the MR deployed and docked at the DS for tool cleaning, disinfection, and sterilization.

2.8 Mechanism of Components

Pre-Surgery and During Surgery: Before a scheduled surgical procedure, the mobile robot deploys from its charging station. The robot's custom lid attachment and claw unlock and detach from its top, lifted by the rail system, specifically the z-rail and z-motor. Subsequently, the mobile robot positions itself near the surgical bed, initiates the procedure protocol, displays it on its main screen, and locks it in a ready-to-use state. Due to its navigation capabilities, the robot can navigate around humans in the room, avoiding potential damage or injury. The navigation system comprises a laser module, a rear proximity sensor, and a front-facing camera. Once the medical team is prepared, the robot activates a timer and enters "operation mode" through voice commands. It displays the active timer, the procedure name, available tools, and an expected usage flow. Surgeons or assistants can request surgical tools via voice commands, and the robot will locate and deliver them. After use, the robot readies the tool for the next handoff. This process continues, and the robot monitors time, workflow, inventory, and potential tool needs.

Post-Surgery: Upon completion, the robot returns to its docking station. The station initiates the disinfection, cleaning, and sanitation modes. To begin disinfection, the rail system lines up the custom claw with the robot's top, and individual elevators rise for cup collection. Once collected, the rail system aligns with the ultrasonic cleaning module, the claw, the cups and custom lid attachments, and the seals for ultrasonic cleaning. Upon completion, the claw unlocks the collection rail system to transfer the tools to the mechanical cleaning module. Here, the claw and lid attachments lock into place for mechanical cleaning. The rail system will then take the claw, cups, and lid to the autoclave for steam sanitation. After completing all three steps, the collection mechanism returns to its initial position on the mobile robot. The individual elevators rise, placing the sterilized surgical equipment and tool cups back onto the robot. The custom lid attachment locks and seals to protect the tools until the next deployment. This means that the robot protects against environmental interference so that potential bacteria or germs in the air do not attach themselves to the cleaned tool's surface.

2.9 Application of REST

The REST system is primarily designed to work autonomously, allowing it to be capable of assisting surgeons during surgery. There are three ways the Mobile Robot can be deployed prior to the surgery: via a physical switch, a voice command, or a pre-programmed schedule. The mobile robot would then deploy autonomously with or without medical personnel in the operating room and park in its designated spot next to the surgical bed.

During surgery: Medical tools can be requested via voice commands; everything can be visually displayed on a screen. For example, if the surgeon needs a Metzenbaum Scissor, the Surgeon can say: "REST, hand me a Metzenbaum Scissor." The robot will then extend its linear slide elevator that holds the cup with the tool to a certain height (which will be the same throughout the surgery) to hand the tool over to the surgeon. The tool in question would be displayed on the visual interface, along with a timer and other useful information for the medical personnel present. A surgeon may want to see what tools are available. They must verbally ask the robot to display available tools on the screen to do this. To do this, a surgeon may say: "REST, what tools are available?".

REST's voice command system supports efficiency in the operating room by enabling hands-free interaction between surgeons and their tools. This innovative feature allows surgeons to request instruments without breaking their focus on the procedure, ensuring seamless and uninterrupted surgical workflows. The system's visual interface complements voice commands by displaying requested tools, timers, and other critical information on a screen. This hands-free interaction minimizes delays and enhances the overall speed and precision of surgical operations.

Post-Surgery: Once the surgery is over, the Mobile Robot would automatically return to its docking station or could be prompted by humans. Upon a successful command, the robot will move to the docking station for tools to be retrieved and cleaned, disinfected, and sanitized. The tools and holding cups at the docking station would undergo a rigorous 3-step sanitizing process. The three-step cleaning process guarantees the highest level of sterility for surgical tools. Ultrasonic cleaning ensures that all debris is thoroughly removed. Mechanical cleaning further eliminates any remaining contaminants, while autoclave sterilization ensures that all bacteria, viruses, and spores are eradicated. This comprehensive approach minimizes the risk of infection and ensures patient safety. Finally, the tools will be loaded back into the mobile robot and sealed off with a custom lid to prevent contamination until the next procedure.

3.0 Results and Discussion

The implementation of the Robotic Essential Surgical Technologist (REST) in the operating room as a non-invasive medical device could result in significant improvements in surgical workflows, patient safety, and healthcare efficiency. The innovative design of REST streamlines surgical procedures by reducing setup and turnover times, minimizing the risk of human error and infection, and alleviating the workload on surgical staff. These enhancements enable medical personnel to focus more on patient care, addressing the pressing issue of labor shortages in healthcare. However, further research is needed to fully understand the potential impact of REST on surgical workflows, patient outcomes, and the roles of surgical staff. The following key points have been identified as potential benefits:

a. Enhanced Infection Control, REST's meticulously designed and rigorously executed cleaning and sterilization procedures between surgeries significantly reduce the risk of cross-contamination and the spread of pathogens. By adhering to a standardized sanitation process, REST serves as a robust defense against lapses in hygiene protocols that may arise from human error. This innovative approach establishes a safer and more reliable healthcare environment, ultimately contributing to improved patient outcomes and reduced healthcare-associated infections.

b. Minimized human error, By automating surgical tool preparation and delivery, REST minimizes the occurrence of human errors, such as providing the wrong tool or inadequate sterilization. This is particularly important given the possibility of fatigue during lengthy surgeries. More than half of surgeries that result in adverse events are caused by human performance deficiencies. REST's consistent and precise performance increases the likelihood of successful surgical outcomes, enhancing patient safety and reducing the risk of complications arising from human mistakes.

c. Addressing healthcare workforce shortages, REST's automation capabilities help alleviate the impact of labor shortages in healthcare by reducing the reliance on human surgical technologists. As a reliable and efficient alternative to human staff, REST ensures that surgical procedures can continue uninterrupted, even in the face of workforce challenges. This is particularly valuable in regions where skilled healthcare professionals are in short supply, enabling healthcare facilities to maintain high-quality care despite staffing limitations. Furthermore, REST can address the projected increase in healthcare demand as the need for healthcare workers is expected to surpass supply by 2025. By automating key tasks, REST allows healthcare facilities to scale operations more efficiently, meeting the rising demand for surgical procedures without proportionally increasing human staffing requirements.

d. Scalable solution, The success of more than 12 million procedures performed using intuitive robotic systems highlights REST's potential to enhance surgical efficiency and outcomes on a global scale. REST's standardized design and use of readily available materials facilitate large-scale production and global deployment, making it a scalable solution for robotic-assisted surgeries. The universal nature of REST's design allows for seamless integration into most surgical and operating rooms, reducing the need for extensive modifications or specialized infrastructure. This scalability enables healthcare facilities around the world to adopt REST, regardless of their location or resource constraints, ultimately expanding access to high-quality, efficient surgical care.

e. Streamlined pre-operative preparation, REST autonomously prepares surgical tools in advance of surgeries, allowing medical personnel to focus on other critical tasks and reducing patient wait times. By efficiently handling the tool preparation process, REST optimizes the pre-operative workflow, ensuring that surgeries can commence promptly and without unnecessary delays. This streamlined approach improves overall surgical efficiency and enhances the patient experience.

f. Integrated sterilization process, REST incorporates the industry-standard 3-step cleaning process, which includes ultrasonic cleaning, mechanical cleaning, and autoclave cleaning. These methods are performed autonomously within the three separate chambers of REST's Docking Station, each specializing in one of the cleaning methods. The tools are efficiently moved between these chambers using a mobile XY-axis system, providing a quick, cost-effective, and autonomous solution for thorough tool sterilization.

g. Non-invasive surgical assistance, REST is designed to assist in the operating room without performing robotic-assisted surgery, alleviating concerns about machines performing invasive procedures. By focusing on non-invasive tasks, such as tool preparation and delivery, REST supports surgeons and surgical staff without directly interacting with patients. This emphasis on assistive functionalities ensures that REST remains a valuable tool for enhancing surgical efficiency and safety while respecting the expertise and judgment of human surgeons.

h. Autonomous operation and low maintenance, REST's fully autonomous functionality and minimal maintenance requirements contribute to its efficiency and reliability in the operating room. Once integrated into the surgical workflow, REST operates independently, requiring little human intervention or oversight. This autonomous operation frees up valuable time for surgical staff, allowing them to focus on more critical aspects of patient care. Additionally, REST's robust design and use of durable materials minimize the need for frequent maintenance, ensuring optimal performance and reducing downtime.

i. Enhanced surgical room navigation, REST's mobile design enables efficient navigation within standard operating and surgical rooms, making it easily accessible to surgeons without extensive setup. The robot's compact size and agile movement allow it to maneuver seamlessly around the surgical environment, adapting to the unique layout of each operating room. With an average operating room size of 400-600 square feet, REST can navigate this space quickly and efficiently, potentially reducing movement time compared to human staff. This flexibility ensures that REST can be readily integrated into existing surgical facilities, minimizing disruption to established workflows and maximizing its potential for widespread adoption.

j. Surgical technologist replacement, REST serves as a cost-effective and efficient alternative to extensively trained surgical technicians. By automating the tasks typically performed by surgical technologists, REST bypasses the need for years of training and hands-on experience. This not only reduces the reliance on human staff but also ensures a consistent and standardized level of performance, mitigating the variability associated with human technologists.

k. Improved surgical room workflow, REST's compact design and efficient tool turnaround time contribute to reducing congestion and optimizing the overall efficiency of the surgical environment. By streamlining the tool preparation and delivery process, REST minimizes the need for human movement within the operating room, reducing the risk of contamination and enhancing overall hygiene. Moreover, the rapid turnaround of sterilized tools between surgeries facilitated by REST's docking station further optimizes surgical workflow, allowing for more procedures to be performed in a given timeframe.

l. Cost-effectiveness, REST provides a cost-effective solution for surgical assistance and tool management. As a one-time investment, REST eliminates the ongoing costs associated with employing human surgical assistants or technologists. Moreover, REST's efficient operation and ability to streamline surgical workflows can lead to cost savings by reducing the duration of surgeries, minimizing the risk of complications, and increasing the number of procedures that can be performed in a given timeframe. These cost benefits make REST an attractive option for healthcare facilities looking to optimize their resources and improve financial sustainability.

The implementation of REST in surgical environments also presents challenges that need to be considered. The following key points have been identified as potential drawbacks:

a. Initial Cost, The initial investment in REST technology can be substantial. This includes the cost of the robotic system itself and the necessary infrastructure upgrades to accommodate it.

b. Training Requirements, Surgical staff will require training to effectively operate and interact with REST. This retraining may temporarily disrupt normal operations and incur additional costs. Additionally, as REST assumes many tasks traditionally performed by surgical technologists, these professionals may need to be reallocated to other aspects of surgical flow.

c. Maintenance and Upkeep, Regular maintenance of the robotic system is essential to ensure its optimal performance. This includes routine checks, software updates, and potential repairs, which can add to the operational costs.

4.0 Conclusion

The Robotic Essential Surgical Technologist (REST) is a novel approach in non-invasive medical robotics that enhances the efficiency, safety, and quality of surgical procedures. By automating critical tasks typically performed by human surgical technologists, REST streamlines workflows, reduces setup and turnover times, and minimizes the risk of human error and infection. The system's innovative design, which includes a mobile robot for precise tool delivery and a docking station for autonomous cleaning, disinfection, and sterilization, establishes a new standard in surgical assistance. REST addresses pressing challenges faced by healthcare systems worldwide, such as labor shortages, by potentially alleviating the workload on surgical staff and enabling them to focus more on patient care. As a cost-effective and scalable solution, REST has the potential to transform surgical practices and shape the future of robotic assistance in healthcare.

While the introduction of REST represents a significant development, further research is needed to fully understand its impact on surgical workflows, patient outcomes, and the roles of surgical staff. As the technology evolves and is implemented in real-world settings, it is essential to assess its performance, gather feedback from healthcare professionals, and make necessary improvements to optimize its functionality and usability. Nonetheless, innovations like REST serve as a testament to the power of technology in advancing healthcare delivery and improving the lives of patients worldwide.
